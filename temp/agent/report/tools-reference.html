Chi tiết Hàm của Tools & Skills

Báo cáo chi tiết cấu trúc mã nguồn, tham số đầu vào/đầu ra, thuật toán và giải thích logic cho từng hàm trong lớp công cụ (Tools) và lớp kỹ năng nghiệp vụ (Skills).

1. Tool: search_vector_db.py

Nằm tại đường dẫn: tools/search_vector_db.py. Đây là tệp tin chịu trách nhiệm thực hiện tìm kiếm tương đồng ngữ nghĩa (semantic search) thông qua vector và định vị điểm neo đồ thị.

Hàm: search_vector_db(query_text, limit, provider)

Chức năng: Nhận câu hỏi ngôn ngữ tự nhiên từ người dùng, sinh vector embedding tương ứng, thực hiện so khớp cosine trên Neo4j Vector Index và JOIN ngược để lấy metadata nguồn trích dẫn.

  • Tham số đầu vào:
    • query_text (str): Câu hỏi hoặc đoạn truy vấn thô của người dùng.
    • limit (int, mặc định = 5): Số lượng kết quả liên quan nhất cần trả về.
    • provider (str): OpenAI hoặc Gemini (dùng để chọn mô hình sinh embedding tương thích).
  • Dữ liệu trả về (Output Schema): Danh sách các dictionary chứa:
    • unit_id (str): ID duy nhất của node văn bản luật (TextUnit).
    • accumulated_text (str): Nội dung văn bản luật chi tiết.
    • citation (str): Trích dẫn cụ thể (ví dụ: "Điều 10").
    • comp_id (str): ID của điều khoản Component (dùng làm điểm neo đồ thị).
    • norm_number (str): Số hiệu văn bản luật (ví dụ: "103/2014/TT-BTC").
    • norm_type (str): Loại văn bản luật (ví dụ: "Thông tư").
    • score (float): Điểm số độ tương đồng vector (0.0 -> 1.0).

Truy vấn Cypher thực tế:

CALL db.index.vector.queryNodes('textunit_embedding_index', $limit, $query_vector)
YIELD node, score
OPTIONAL MATCH (c:Component)-[:HAS_TEXTUNIT]->(node)
OPTIONAL MATCH (n:Norm)-[:CONTAINS*1..6]->(c)
RETURN node.unit_id AS unit_id, 
       node.accumulated_text AS accumulated_text, 
       c.citation AS citation,
       c.comp_id AS comp_id,
       n.norm_number AS norm_number,
       n.norm_type AS norm_type,
       score

Giải thích chi tiết truy vấn:

  1. Tìm kiếm Vector: CALL db.index.vector.queryNodes quét index textunit_embedding_index dựa trên vector câu hỏi $query_vector để chọn ra các node TextUnit sát nghĩa nhất.
  2. JOIN Điều khoản (Component): OPTIONAL MATCH (c:Component)-[:HAS_TEXTUNIT]->(node) ánh xạ ngược đoạn văn bản về node điều khoản cha của nó để lấy tiêu đề trích dẫn và ID điều khoản. Dùng OPTIONAL MATCH để đảm bảo không bị mất kết quả nếu dữ liệu bị khuyết liên kết.
  3. JOIN Văn bản gốc (Norm): OPTIONAL MATCH (n:Norm)-[:CONTAINS*1..6]->(c) đi ngược từ điều khoản lên văn bản luật gốc. Đường dẫn đệ quy CONTAINS*1..6 đi xuyên qua các lớp phân cấp trung gian (Chương, Mục, Điều, Khoản...) để tìm đúng số hiệu văn bản (norm_number).
  4. Bảo vệ lỗi (Robustness): Toàn bộ logic được bao bọc trong khối try-except. Nếu kết nối CSDL Neo4j hoặc API Embedding bị lỗi, hàm tự động in log lỗi và trả về danh sách rỗng [] an toàn để tránh làm crash LangGraph loop.

2. Tool: search_legal_graph.py

Nằm tại đường dẫn: tools/search_legal_graph.py. Tệp tin này chịu trách nhiệm thực thi các truy vấn Cypher duyệt đồ thị tri thức để phát hiện liên kết chéo và các chuỗi sửa đổi pháp luật.

Hàm: get_norm_relations(code)

Chức năng: Tra cứu toàn bộ các văn bản luật có quan hệ liên kết chéo với một văn bản đích (ví dụ: Nghị định hướng dẫn Luật, hoặc Thông tư sửa đổi Nghị định).

  • Tham số đầu vào: code (str): Số hiệu văn bản luật cần tra cứu (ví dụ: "83/2014/NĐ-CP").
  • Dữ liệu trả về: Danh sách các dictionary chứa:
    • rel_type (str): Tên mối quan hệ (ví dụ: AMENDS, IMPLEMENTS).
    • code (str): Số hiệu văn bản liên kết.
    • type (str): Loại văn bản liên kết.
    • direction (str): Chiều quan hệ (outgoing - xuôi từ văn bản này đi, hoặc incoming - ngược từ văn bản khác tác động vào).

Truy vấn Cypher thực tế:

# Truy vấn chiều xuôi (outgoing)
MATCH (n:Norm) WHERE n.norm_number = $code
MATCH (n)-[r:CITES|IMPLEMENTS|SUPPLEMENTS|AMENDS|PARTIALLY_TERMINATES|TERMINATES|REFERS_TO|RELATED_TO]->(target:Norm)
RETURN type(r) AS rel_type, target.norm_number AS code, target.norm_type AS type, 'outgoing' AS direction

# Truy vấn chiều ngược (incoming)
MATCH (n:Norm) WHERE n.norm_number = $code
MATCH (source:Norm)-[r:CITES|IMPLEMENTS|SUPPLEMENTS|AMENDS|PARTIALLY_TERMINATES|TERMINATES|REFERS_TO|RELATED_TO]->(n)
RETURN type(r) AS rel_type, source.norm_number AS code, source.norm_type AS type, 'incoming' AS direction

Giải thích chi tiết truy vấn:

  1. Chiều xuôi (outgoing): Tìm các văn bản mà văn bản hiện tại trích dẫn (CITES), tham chiếu (REFERS_TO), sửa đổi (AMENDS) hoặc hướng dẫn (IMPLEMENTS).
  2. Chiều ngược (incoming): Tìm các văn bản mới hơn đang tác động lên văn bản hiện tại (ví dụ: Nghị định mới bãi bỏ một phần Nghị định cũ).
  3. Hỗ trợ đủ 8 mối quan hệ: Đảm bảo không bỏ sót bất kỳ mối liên kết vật lý nào giữa các Norm trong DB thực tế.
  4. Bảo vệ lỗi: Bao bọc bằng try-except trả về danh sách rỗng [] nếu kết nối Neo4j bị ngắt quãng.

Hàm: get_component_actions(comp_id)

Chức năng: Tìm kiếm lịch sử sửa đổi của một điều luật (Component) qua chuỗi hành động sửa đổi bắc cầu (đệ quy) hoặc danh sách điều khoản dẫn chiếu chéo (flat fallback).

  • Tham số đầu vào: comp_id (str): ID duy nhất của điều khoản (ví dụ: "127429__c59").
  • Dữ liệu trả về: Danh sách lịch sử hành động sửa đổi/dẫn chiếu (chứa amending_doc, action_type, action_text, depth).

Truy vấn Cypher thực tế:

# Cách 1: Duyệt đệ quy chuỗi sửa đổi (Action Node)
MATCH path = (a:Action)-[:APPLY_TO|HAS_ACTION*1..6]->(c:Component {comp_id: $comp_id})
OPTIONAL MATCH (a)-[:HAS_TEXTUNIT]->(t:TextUnit)
RETURN a.amending_doc_number AS amending_doc, 
       a.relation_type AS action_type, 
       t.accumulated_text AS action_text,
       t.unit_id AS unit_id,
       length(path) AS depth
ORDER BY depth ASC

# Cách 2 (Fallback phẳng): Dẫn chiếu chéo REFERS_TO
MATCH (c:Component {comp_id: $comp_id})-[r:REFERS_TO]->(target:Component)
OPTIONAL MATCH (target)-[:HAS_TEXTUNIT]->(t:TextUnit)
RETURN target.comp_id AS target_comp_id, 
       target.citation AS citation, 
       t.accumulated_text AS text

Giải thích chi tiết truy vấn:

  1. Duyệt đệ quy chuỗi sửa đổi: [:APPLY_TO|HAS_ACTION*1..6] cho phép đi xuyên qua nhiều tầng sửa đổi trung gian để tìm ra sửa đổi mới nhất (độ sâu lớn nhất). Chiều đi đúng chuẩn trong DB là từ node `Action` hướng tới `Component` qua quan hệ APPLY_TO.
  2. Sắp xếp theo độ sâu: ORDER BY depth ASC sắp xếp kết quả sửa đổi từ trực tiếp đến bắc cầu để Agent dễ theo dõi diễn tiến pháp lý.
  3. Fallback dẫn chiếu chéo: Nếu điều khoản không bị sửa đổi bởi luật nào (danh sách actions trống), hệ thống thực hiện tìm kiếm các điều khoản mà điều này dẫn chiếu tới (quan hệ REFERS_TO) để lấy nội dung bổ trợ.

Hàm: get_component_context(comp_id)

Chức năng: Lấy thông tin tiêu đề điều luật, cấp bậc của nó, và tìm văn bản gốc chứa nó để làm ngữ cảnh nền tảng cho Agent lập luận.

  • Tham số đầu vào: comp_id (str): ID duy nhất của điều khoản.
  • Dữ liệu trả về: Dictionary chứa thông tin norm (văn bản gốc) và component (tiêu đề, trích dẫn, cấp bậc).

Truy vấn Cypher thực tế:

# 1. Lấy văn bản gốc chứa Component
MATCH (n:Norm)-[:CONTAINS*1..6]->(c:Component {comp_id: $comp_id})
RETURN n.norm_number AS code, n.norm_type AS type LIMIT 1

# 2a. Thử tìm qua Version (Nâng cao)
MATCH (c:Component {comp_id: $comp_id})-[:HAS_VERSION]->(v:Version)
RETURN c.title_text AS title, c.citation AS citation, c.level AS level, v.version_id AS version_id LIMIT 1

# 2b. Fallback tìm trực tiếp Component (Phẳng)
MATCH (c:Component {comp_id: $comp_id})
RETURN c.title_text AS title, c.citation AS citation, c.level AS level LIMIT 1

Giải thích chi tiết truy vấn:

  1. Xác định văn bản gốc: Quan hệ CONTAINS liên kết trực tiếp từ Norm mẹ tới Component con cấp 1. Lấy số hiệu văn bản làm căn cứ nguồn.
  2. Fallback tương thích: Ưu tiên lấy thông tin điều khoản đi kèm ID phiên bản Version. Nếu DB bị thiếu dữ liệu Version, hệ thống tự động fallback sang truy vấn Component phẳng để không làm gián đoạn việc hiển thị dữ liệu của Agent.
  3. Bảo vệ lỗi: Trả về cấu trúc mặc định an toàn {"norm": {}, "component": {}} khi xảy ra lỗi kết nối.

3. Tool: web_search_mcp.py

Nằm tại đường dẫn: tools/web_search_mcp.py. Tệp tin này cung cấp khả năng tìm kiếm internet fallback khi CSDL GraphRAG nội bộ không chứa đủ ngữ cảnh hoặc cần thông tin cập nhật mới nhất.

Hàm: web_search_mcp(query, max_results, limit_to_trusted, search_depth)

Chức năng: Truy vấn Tavily Search API (qua HTTP REST Client tích hợp sẵn) để tìm kiếm thông tin bổ trợ. Mặc định tự động giới hạn tìm kiếm trong tập hợp các tên miền tin cậy và tự động **loại bỏ kết quả trùng lặp URL** để tối ưu hóa lượng token gửi đến Agent.

  • Tham số đầu vào:
    • query (str): Từ khóa tìm kiếm hoặc câu hỏi thô.
    • max_results (int, mặc định = 5): Số lượng trang kết quả tối đa cần lấy.
    • limit_to_trusted (bool, mặc định = True): Quyết định có lọc kết quả theo danh sách website uy tín hay không.
    • search_depth (str, mặc định = "basic"): Độ sâu tìm kiếm của Tavily (chấp nhận "basic" hoặc "advanced" cho tra cứu chuyên sâu).
  • Danh sách tên miền tin cậy (Trusted Domains):
    • vbpl.vn (Cơ sở dữ liệu quốc gia về văn bản pháp luật)
    • thuvienphapluat.vn (Cơ sở dữ liệu tư nhân lớn về luật)
    • chinhphu.vn (Cổng thông tin điện tử Chính phủ Việt Nam)
    • mof.gov.vn (Cổng thông tin Bộ Tài chính)
    • gdt.gov.vn (Cổng thông tin Tổng cục Thuế)
  • Dữ liệu trả về (Output Schema): Danh sách các dictionary chứa:
    • title (str): Tiêu đề của trang kết quả tìm kiếm.
    • link (str): Đường dẫn liên kết nguồn (URL).
    • snippet (str): Nội dung tóm tắt ngắn từ trang web.
    • trusted (bool): Trạng thái xác định kết quả thuộc tập tên miền tin cậy hay không.
  • Cấu hình Proxy (Tùy chọn):
    • Hệ thống tự động đọc biến môi trường WEB_SEARCH_PROXY từ tệp .env (Ví dụ: http://user:pass@ip:port).
    • Nếu giá trị này trống hoặc không tồn tại, hệ thống tự động fallback sử dụng kết nối mạng trực tiếp, bảo đảm tính tiện dụng khi phát triển local.

Cơ chế Fallback & Tự phản tỉnh (Self-Reflection):

  1. Fallback 1 (Khi lỗi kết nối hoặc rate limit lọc): Nếu việc truy vấn kèm bộ lọc bị Tavily từ chối hoặc lỗi mạng, hàm tự động bắt exception và thử lại bằng cách gọi đệ quy với limit_to_trusted=False để lấy kết quả tự do không giới hạn tên miền.
  2. Fallback 2 (Khi lọc không có kết quả): Nếu tìm kiếm có lọc chạy thành công nhưng không trả về bất kỳ kết quả nào, hệ thống tự động kích hoạt tìm kiếm tự do không giới hạn tên miền trên Tavily để đảm bảo Agent luôn có dữ liệu thô.
  3. Bảo vệ an toàn vòng lặp: Nếu cả hai lần tìm kiếm đều thất bại, hàm trả về danh sách rỗng [] và in cảnh báo log để tránh gây crash cho toàn bộ AgentStateGraph.

4. Tool: check_legal_status.py

Nằm tại đường dẫn: tools/check_legal_status.py. Tệp tin này kiểm tra trạng thái hiệu lực pháp lý thực tế của một văn bản pháp luật (Norm) từ Neo4j, đồng thời tìm các văn bản sửa đổi/bãi bỏ liên quan.

Hàm: check_legal_status(code)

Chức năng: Nhận số hiệu văn bản (code), thực hiện truy vấn Neo4j để lấy trạng thái hiệu lực hiện tại, danh sách văn bản bãi bỏ/thế chỗ (TERMINATES/PARTIALLY_TERMINATES) và sửa đổi/bổ sung (AMENDS/SUPPLEMENTS).

  • Tham số đầu vào:
    • code (str): Số hiệu văn bản luật cần kiểm tra (ví dụ: "65/2013/NĐ-CP").
  • Dữ liệu trả về (Output Schema): Dictionary chứa:
    • code (str): Số hiệu văn bản kiểm tra.
    • exists (bool): Xác định văn bản có tồn tại trong CSDL hay không.
    • type (str): Loại văn bản (Nghị định, Luật, Thông tư...).
    • validity_status (str): Trạng thái hiệu lực (Còn hiệu lực, Hết hiệu lực...).
    • terminated_by (List[Dict]): Danh sách văn bản bãi bỏ hoàn toàn/một phần.
    • amended_by (List[Dict]): Danh sách văn bản sửa đổi/bổ sung.
    • error (str hoặc None): Chi tiết thông báo lỗi nếu xảy ra sự cố truy vấn CSDL.

Các câu truy vấn Cypher:

# 1. Trạng thái cơ bản
MATCH (n:Norm) WHERE n.norm_number = $code
RETURN n.norm_number AS code, n.norm_type AS type, n.validity_status AS validity_status LIMIT 1

# 2. Các văn bản bãi bỏ
MATCH (n:Norm) WHERE n.norm_number = $code
MATCH (terminator:Norm)-[r:TERMINATES|PARTIALLY_TERMINATES]->(n)
RETURN terminator.norm_number AS code, terminator.norm_type AS type, type(r) AS rel_type

# 3. Các văn bản sửa đổi, bổ sung
MATCH (n:Norm) WHERE n.norm_number = $code
MATCH (amender:Norm)-[r:AMENDS|SUPPLEMENTS]->(n)
RETURN amender.norm_number AS code, amender.norm_type AS type, type(r) AS rel_type

Cập nhật mới (Changelog):

Bảo vệ lỗi toàn diện (try-except): Toàn bộ logic kết nối và truy vấn CSDL được bao bọc trong khối try-except để trả về dictionary trạng thái mặc định an toàn khi có sự cố kết nối Neo4j, tránh gây crash cho AgentStateGraph. Trường error được bổ sung để truyền đạt lỗi cụ thể.

5. Tool: validate_citations.py

Nằm tại đường dẫn: tools/validate_citations.py. Tệp tin này dùng để đối chiếu và kiểm chứng độ chính xác của các trích dẫn pháp luật trong câu trả lời dự thảo của Agent với nội dung gốc trong CSDL Neo4j bằng Cosine Similarity, chống ảo tưởng (hallucination).

Hàm: validate_citations(draft_answer, provider)

Chức năng: Trích xuất các trích dẫn dạng [Văn bản - Điều khoản] bằng Regex, truy vấn Neo4j lấy văn bản gốc, sinh Vector Embedding cho ngữ cảnh của trích dẫn và nội dung gốc để so khớp độ tương đồng Cosine.

  • Tham số đầu vào:
    • draft_answer (str): Phản hồi dự thảo chứa các trích dẫn cần kiểm chứng.
    • provider (str): OpenAI hoặc Gemini để chọn mô hình sinh embedding tương ứng.
  • Dữ liệu trả về (Output Schema): Danh sách các kết quả kiểm chứng trích dẫn gồm:
    • raw_citation (str): Chuỗi trích dẫn thô khớp được (VD: [65/2013/NĐ-CP - Điều 12]).
    • exists (bool): Xác định văn bản và điều khoản trích dẫn có tồn tại trong CSDL hay không.
    • valid (bool): Xác định trích dẫn có hợp lệ hay không (phản ánh việc tìm thấy đúng văn bản và điều khoản trong CSDL).
    • score (float): Điểm tương đồng Cosine (0.0 -> 1.0).
    • correct_text (str): Nội dung điều khoản gốc trong CSDL để đối chiếu.
    • reason (str): Lý do chi tiết trích dẫn chính xác hoặc bị sai lệch, hoặc lỗi.
    • error (str hoặc None): Chi tiết thông báo lỗi nếu xảy ra sự cố trong quá trình sinh embedding hoặc kết nối DB.

Truy vấn Cypher thực tế:

MATCH (n:Norm) WHERE n.norm_number = $norm_code
MATCH (n)-[:CONTAINS*1..6]->(c:Component)
WHERE c.citation = $citation_name
OPTIONAL MATCH (c)-[:HAS_TEXTUNIT]->(t:TextUnit)
RETURN c.comp_id AS comp_id, 
       t.accumulated_text AS text, 
       t.accumulated_text AS accumulated_text, 
       t.embedding AS embedding
LIMIT 1

Cập nhật mới (Changelog):

  • Sửa lỗi truy vấn phân cấp (Hierarchical Fix): Thay đổi mối quan hệ MATCH Component từ [:CONTAINS] đơn tầng thành [:CONTAINS*1..6] để đi xuyên qua cấu trúc Chương, Mục, Điều khoản nằm sâu trong văn bản luật gốc.
  • Bảo vệ lỗi kết nối (try-except): Bao bọc toàn bộ phiên làm việc của Neo4j và logic DB trong khối try-except để trả về kết quả không hợp lệ kèm lý do lỗi kết nối thay vì làm sập luồng LangGraph.
  • Thay đổi ý nghĩa trường valid: Trường valid hiện phản ánh trực tiếp trạng thái tìm thấy văn bản và điều khoản trong CSDL, trong khi điểm số tương đồng Cosine chỉ là tín hiệu tham khảo để phát hiện diễn giải sai nghĩa.

6. Tool: read_document_chunk.py

Nằm tại đường dẫn: tools/read_document_chunk.py. Tệp tin này chịu trách nhiệm đọc và phân mảnh nội dung tệp tài liệu pháp lý (PDF, Word, Text) để tránh làm phình ngữ cảnh của LLM.

Hàm: read_document_chunk(file_path, chunk_index, chunk_size)

Chức năng: Đọc tệp tin tài liệu dựa trên phần mở rộng, phân mảnh văn bản theo kích thước ký tự và trích xuất mảnh dữ liệu tại vị trí chỉ định.

  • Tham số đầu vào:
    • file_path (str): Đường dẫn tuyệt đối hoặc tương đối đến tệp tài liệu.
    • chunk_index (int, mặc định = 0): Vị trí mảnh cần đọc (0-indexed).
    • chunk_size (int, mặc định = 2000): Kích thước ký tự tối đa của mảnh văn bản.
  • Các định dạng hỗ trợ:
    • .txt: Đọc tệp tin văn bản thuần qua hàm hỗ trợ read_text_file(file_path).
    • .docx: Đọc tệp Word qua hàm read_docx_file(file_path) (sử dụng thư viện python-docx).
    • .pdf: Đọc tệp PDF qua hàm read_pdf_file(file_path) (sử dụng thư viện pypdf).
  • Dữ liệu trả về (Output Schema): Dictionary chứa:
    • file_name (str): Tên tệp tin thô được trích xuất từ đường dẫn.
    • chunk_text (str): Nội dung văn bản của mảnh hiện tại.
    • chunk_index (int): Chỉ số mảnh thực tế được trả về (sau khi kiểm tra biên).
    • total_chunks (int): Tổng số phân mảnh được tính toán dựa trên kích thước tệp.
    • total_characters (int): Tổng độ dài ký tự của văn bản gốc trong tài liệu.

Bảo vệ lỗi và Các ràng buộc:

  • Ràng buộc kích thước: Hàm ném ra lỗi ValueError("chunk_size phải lớn hơn 0.") nếu tham số kích thước không hợp lệ.
  • Kiểm tra tồn tại: Ném lỗi FileNotFoundError nếu đường dẫn tệp tin không tồn tại.
  • Định dạng không hỗ trợ: Ném lỗi ValueError nếu định dạng tệp tin khác .txt, .docx, .pdf.
  • Bảo vệ vượt biên: Tự động điều chỉnh chunk_index về 0 nếu nhỏ hơn 0, hoặc về mảnh cuối cùng total_chunks - 1 nếu vượt quá tổng số mảnh thực tế.

7. Tool: generate_report.py

Nằm tại đường dẫn: tools/generate_report.py. Tệp tin này định nghĩa Strategy Pattern để xuất bản báo cáo tư vấn pháp lý dưới nhiều định dạng khác nhau.

Hàm: generate_report(data, format_type)

Chức năng: Hàm wrapper nhận dữ liệu tư vấn đã hoàn chỉnh và áp dụng formatter tương ứng với kiểu định dạng yêu cầu để xuất ra chuỗi báo cáo.

  • Tham số đầu vào:
    • data (Dict[str, Any]): Dữ liệu báo cáo chứa các cấu trúc: query, summary, details (danh sách phân tích chi tiết), conflicts (danh sách xung đột) và citations (danh sách trích dẫn).
    • format_type (str, mặc định = "markdown"): Kiểu định dạng xuất báo cáo (chấp nhận markdown, md, html).
  • Cấu trúc thiết kế (Strategy Pattern):
    • BaseReportFormatter: Lớp cơ sở trừu tượng quy định giao diện format(data).
    • MarkdownFormatter: Định dạng dữ liệu thành chuỗi Markdown tiêu chuẩn, có định dạng rõ ràng, thụt dòng blockquote cho trích dẫn nguồn.
    • HTMLFormatter: Định dạng thành tài liệu HTML5 hoàn chỉnh tích hợp sẵn CSS nội bộ (Inline CSS) với màu sắc chủ đạo đồng nhất, khối cảnh báo xung đột nổi bật.
    • PDFFormatter & ExcelFormatter: Đã được định nghĩa lớp nhưng ném ra NotImplementedError để chờ triển khai nâng rộng.
  • Dữ liệu trả về (Output Schema): Chuỗi văn bản (string) đã được định dạng tương ứng với style được lựa chọn.

Bảo vệ lỗi:

Hàm wrapper tự động kiểm tra loại định dạng đầu vào. Nếu format_type không được hỗ trợ trong danh mục (ví dụ chọn pdf hoặc excel hiện tại), hàm sẽ ném ra lỗi ValueError kèm thông điệp hướng dẫn rõ ràng: "Không hỗ trợ định dạng báo cáo: {format_type}. Hiện chỉ hỗ trợ: markdown, md, html.".

8. Tool: send_email_action.py

Nằm tại đường dẫn: tools/send_email_action.py. Tệp tin này thực hiện tác vụ gửi email báo cáo kết quả tư vấn pháp lý cho người dùng thông qua giao thức SMTP bảo mật.

Hàm: send_email_action(recipient_email, subject, body_text, attachment_path)

Chức năng: Đóng gói nội dung email dưới dạng MIMEMultipart, đính kèm tệp tin báo cáo (nếu có), thực hiện bắt tay TLS và đăng nhập SMTP Server để gửi thư đi.

  • Tham số đầu vào:
    • recipient_email (str): Địa chỉ email của người nhận.
    • subject (str): Tiêu đề thư.
    • body_text (str): Nội dung văn bản thuần của email.
    • attachment_path (str, tùy chọn): Đường dẫn tệp tin đính kèm.
  • Cấu hình môi trường sử dụng:
    • SMTP_SERVER (mặc định: smtp.gmail.com)
    • SMTP_PORT (mặc định: 587)
    • SMTP_USERNAME & SMTP_PASSWORD: Thông tin tài khoản gửi email.
    • SMTP_SIMULATION (bool): Đọc từ biến môi trường. Nếu được đặt thành true, hệ thống tự động chạy chế độ giả lập để kiểm thử luồng mà không cần kết nối mạng.
  • Dữ liệu trả về (Output Schema): Dictionary chứa:
    • status (str): Trạng thái gửi (sent | simulated | failed).
    • message (str): Thông báo mô tả kết quả chi tiết.
    • recipient (str): Email người nhận.
    • subject (str): Tiêu đề thư đã xử lý.
    • attachment (str, tùy chọn): Đường dẫn tệp tin đính kèm nếu có.

Cơ chế xử lý lỗi và Giả lập an toàn:

  1. Chế độ giả lập: Khi biến SMTP_SIMULATION=true được cấu hình, hàm tự động in thông tin log giả lập ra console và trả về status "simulated" ngay lập tức.
  2. Thiếu cấu hình SMTP: Nếu không ở chế độ giả lập nhưng thiếu cấu hình username/password SMTP, hệ thống in log lỗi và trả về status "failed" hướng dẫn đặt SMTP_SIMULATION=true để chạy thử nghiệm.
  3. Lỗi đính kèm file: Bao bọc logic đọc và mã hóa Base64 tệp đính kèm trong khối try-except riêng. Nếu tệp lỗi hoặc không tìm thấy, hệ thống in cảnh báo "Cannot attach file" nhưng **vẫn tiếp tục gửi email** phần nội dung chữ đi thay vì làm hỏng toàn bộ luồng gửi.
  4. Bảo vệ lỗi SMTP: Toàn bộ quá trình kết nối và gửi thư qua smtplib được bảo vệ bằng try-except, tự động bắt lỗi ghi log và trả về status "failed" cùng lý do cụ thể, tránh làm crash AgentStateGraph.

9. Tool: answer_directly.py

Nằm tại đường dẫn: tools/answer_directly.py. Đây là helper bypass tạo phản hồi trực tiếp bằng LLM dành cho các câu hỏi mang tính chất giao tiếp xã giao (chitchat) hoặc ngoài phạm vi nghiệp vụ pháp lý.

Hàm: answer_directly(query, provider)

Chức năng: Xác định mô hình điều phối tương ứng với nhà cung cấp (Gemini hoặc GPT), thiết lập system instruction định hướng lịch sự để trả lời trực tiếp cho người dùng.

  • Tham số đầu vào:
    • query (str): Đoạn chat giao tiếp thô của người dùng.
    • provider (str): Nhà cung cấp mô hình AI (OpenAI hoặc Gemini).
  • Nguyên tắc định hướng (System Instruction):

    "Bạn là một trợ lý ảo thông minh chuyên về tư vấn pháp luật Việt Nam (như luật Thuế, luật Đất đai). Người dùng hiện tại đang chào hỏi hoặc chitchat/nói chuyện ngoài lề không liên quan đến pháp luật. Hãy phản hồi một cách ngắn gọn, thân thiện, lịch sự và tinh tế hướng người dùng đặt các câu hỏi liên quan đến lĩnh vực tư vấn pháp lý Việt Nam."

  • Dữ liệu trả về (Output Schema): Chuỗi văn bản phản hồi (string) thân thiện, ngắn gọn do LLM tạo ra.

Bảo vệ lỗi (Robustness):

Hàm bao bọc lời gọi sinh văn bản của mô hình bằng khối try-except. Nếu gặp sự cố mất kết nối mạng hoặc lỗi API Key, hàm tự động trả về một câu phản hồi chào hỏi mặc định an toàn: "Xin chào! Tôi có thể giúp gì cho bạn về các vấn đề pháp luật Việt Nam (Thuế, Đất đai...) hôm nay?..." kèm thông tin chi tiết mã lỗi để không làm gián đoạn trải nghiệm người dùng.

10. Lớp Nghiệp vụ Kỹ năng (Skills)

Nằm tại thư mục: skills/. Lớp này đóng gói quy trình gọi các công cụ thô (Tools) ở Tầng 1 và xử lý làm sạch dữ liệu trung gian trước khi cung cấp cho các tác nhân lập luận (Agents) ở Tầng 2.

10.1. Kỹ năng: skills/search_retrieval.py

Chức năng: Đóng gói luồng tra cứu luật đa chiều, kiểm tra hiệu lực thực tế và tự động tìm kiếm bổ trợ trên web nếu CSDL Neo4j GraphRAG thiếu dữ liệu.

  • Hàm thực thi chính: execute_search_retrieval(query_text, provider, limit)
  • Quy trình xử lý (Pipeline):
    1. Gọi search_vector_db để thực hiện tìm kiếm ngữ nghĩa, thu thập các node TextUnit thô và JOIN ngược lấy comp_id, norm_number.
    2. Nếu số lượng kết quả thu được nhỏ hơn limit, tự động gọi web_search_mcp làm fallback để lấy thêm các đoạn thông tin trên internet.
    3. Duyệt qua danh sách văn bản và gọi check_legal_status để xác định hiệu lực pháp lý (còn/hết hiệu lực, các tài liệu thay thế/sửa đổi). Sử dụng cơ chế cache theo norm_number để tránh gọi DB trùng lặp hiệu năng cao.
  • Tham số đầu vào:
    • query_text (str): Câu truy vấn của người dùng.
    • provider (str): Provider của LLM để sinh embedding.
    • limit (int): Giới hạn số lượng kết quả cần lấy.
  • Tham số đầu ra: Danh sách kết quả sạch dạng Dict chứa đầy đủ thông tin: unit_id, comp_id, accumulated_text, citation, norm_number, norm_type, score, source, validity_status, amended_by, terminated_by.

10.2. Kỹ năng: skills/analysis.py

Chức năng: Nghiên cứu sâu ngữ cảnh của một điều luật cụ thể, truy vết đệ quy chuỗi sửa đổi pháp lý và hỗ trợ tích hợp nội dung tài liệu do người dùng tải lên.

  • Hàm thực thi chính: execute_deep_analysis(comp_id, file_path, chunk_index, chunk_size)
  • Quy trình xử lý (Pipeline):
    1. Gọi get_component_context lấy thông tin tiêu đề, cấp bậc điều khoản và văn bản luật gốc.
    2. Gọi get_component_actions để duyệt đệ quy tìm các tài liệu sửa đổi/bãi bỏ liên quan trực tiếp và bắc cầu, hoặc tìm các điều luật dẫn chiếu chéo (REFERS_TO).
    3. Nếu người dùng cung cấp file_path, gọi read_document_chunk để đọc phân đoạn thông tin bổ sung và đính kèm vào ngữ cảnh.
  • Tham số đầu vào:
    • comp_id (str): ID duy nhất của điều khoản (Component).
    • file_path (str, optional): Đường dẫn tệp đính kèm người dùng tải lên.
    • chunk_index (int): Phân đoạn cần đọc của tệp đính kèm.
  • Tham số đầu ra: Dictionary chứa ngữ cảnh hoàn chỉnh: comp_id, norm_context, component_details, modifications, additional_document, error.

10.3. Kỹ năng: skills/reporting.py

Chức năng: Đóng gói quy trình kiểm chứng chéo các trích dẫn chống ảo tưởng, tự động format báo cáo và kích hoạt gửi SMTP email hành động.

  • Hàm thực thi chính: execute_reporting_and_validation(draft_answer, raw_data, provider, format_type, recipient_email, email_subject)
  • Quy trình xử lý (Pipeline):
    1. Gọi validate_citations để trích xuất các trích dẫn dạng [Văn bản - Điều khoản] trong bản nháp, đối chiếu với Neo4j và sinh cosine similarity kiểm tra độ lệch nghĩa.
    2. Tổng hợp thông tin thô từ raw_data và danh sách trích dẫn đã xác thực để gọi generate_report xuất báo cáo theo định dạng yêu cầu (Markdown/HTML).
    3. Nếu có địa chỉ email người nhận (recipient_email), gọi send_email_action để thực hiện gửi email báo cáo (hỗ trợ chế độ simulation an toàn).
  • Tham số đầu vào:
    • draft_answer (str): Nội dung câu trả lời nháp do Agent soạn thảo.
    • raw_data (Dict): Dữ liệu phân tích gốc.
    • format_type (str): Định dạng mong muốn (markdown/html).
    • recipient_email (str, optional): Email người nhận.
  • Tham số đầu ra: Dictionary kết quả: verified_citations (danh sách trích dẫn kèm trạng thái exists/valid), report_content (báo cáo đã được format), email_status (kết quả gửi SMTP), error.
