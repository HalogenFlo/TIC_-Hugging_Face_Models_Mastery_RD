Chi tiết Hàm & Truy vấn của Tools

Báo cáo chi tiết cấu trúc mã nguồn, tham số đầu vào/đầu ra và giải thích truy vấn Cypher cho từng hàm trong lớp công cụ.

1. Tool: search_vector_db.py

Nằm tại đường dẫn: tools/search_vector_db.py. Đây là tệp tin chịu trách nhiệm thực hiện tìm kiếm tương đồng ngữ nghĩa (semantic search) thông qua vector và định vị điểm neo đồ thị.

Hàm: search_vector_db(query_text, limit, provider)

Chức năng: Nhận câu hỏi ngôn ngữ tự nhiên từ người dùng, sinh vector embedding tương ứng, thực hiện so khớp cosine trên Neo4j Vector Index và JOIN ngược để lấy metadata nguồn trích dẫn.

  • Tham số đầu vào:
    • query_text (str): Câu hỏi hoặc đoạn truy vấn thô của người dùng.
    • limit (int, mặc định = 5): Số lượng kết quả liên quan nhất cần trả về.
    • provider (str): OpenAI hoặc Gemini (dùng để chọn mô hình sinh embedding tương thích).
  • Dữ liệu trả về (Output Schema): Danh sách các dictionary chứa:
    • unit_id (str): ID duy nhất của node văn bản luật (TextUnit).
    • accumulated_text (str): Nội dung văn bản luật chi tiết.
    • citation (str): Trích dẫn cụ thể (ví dụ: "Điều 10").
    • comp_id (str): ID của điều khoản Component (dùng làm điểm neo đồ thị).
    • norm_number (str): Số hiệu văn bản luật (ví dụ: "103/2014/TT-BTC").
    • norm_type (str): Loại văn bản luật (ví dụ: "Thông tư").
    • score (float): Điểm số độ tương đồng vector (0.0 -> 1.0).

Truy vấn Cypher thực tế:

CALL db.index.vector.queryNodes('textunit_embedding_index', $limit, $query_vector)
YIELD node, score
OPTIONAL MATCH (c:Component)-[:HAS_TEXTUNIT]->(node)
OPTIONAL MATCH (n:Norm)-[:CONTAINS*1..6]->(c)
RETURN node.unit_id AS unit_id, 
       node.accumulated_text AS accumulated_text, 
       c.citation AS citation,
       c.comp_id AS comp_id,
       n.norm_number AS norm_number,
       n.norm_type AS norm_type,
       score

Giải thích chi tiết truy vấn:

  1. Tìm kiếm Vector: CALL db.index.vector.queryNodes quét index textunit_embedding_index dựa trên vector câu hỏi $query_vector để chọn ra các node TextUnit sát nghĩa nhất.
  2. JOIN Điều khoản (Component): OPTIONAL MATCH (c:Component)-[:HAS_TEXTUNIT]->(node) ánh xạ ngược đoạn văn bản về node điều khoản cha của nó để lấy tiêu đề trích dẫn và ID điều khoản. Dùng OPTIONAL MATCH để đảm bảo không bị mất kết quả nếu dữ liệu bị khuyết liên kết.
  3. JOIN Văn bản gốc (Norm): OPTIONAL MATCH (n:Norm)-[:CONTAINS*1..6]->(c) đi ngược từ điều khoản lên văn bản luật gốc. Đường dẫn đệ quy CONTAINS*1..6 đi xuyên qua các lớp phân cấp trung gian (Chương, Mục, Điều, Khoản...) để tìm đúng số hiệu văn bản (norm_number).
  4. Bảo vệ lỗi (Robustness): Toàn bộ logic được bao bọc trong khối try-except. Nếu kết nối CSDL Neo4j hoặc API Embedding bị lỗi, hàm tự động in log lỗi và trả về danh sách rỗng [] an toàn để tránh làm crash LangGraph loop.

2. Tool: search_legal_graph.py

Nằm tại đường dẫn: tools/search_legal_graph.py. Tệp tin này chịu trách nhiệm thực thi các truy vấn Cypher duyệt đồ thị tri thức để phát hiện liên kết chéo và các chuỗi sửa đổi pháp luật.

Hàm: get_norm_relations(code)

Chức năng: Tra cứu toàn bộ các văn bản luật có quan hệ liên kết chéo với một văn bản đích (ví dụ: Nghị định hướng dẫn Luật, hoặc Thông tư sửa đổi Nghị định).

  • Tham số đầu vào: code (str): Số hiệu văn bản luật cần tra cứu (ví dụ: "83/2014/NĐ-CP").
  • Dữ liệu trả về: Danh sách các dictionary chứa:
    • rel_type (str): Tên mối quan hệ (ví dụ: AMENDS, IMPLEMENTS).
    • code (str): Số hiệu văn bản liên kết.
    • type (str): Loại văn bản liên kết.
    • direction (str): Chiều quan hệ (outgoing - xuôi từ văn bản này đi, hoặc incoming - ngược từ văn bản khác tác động vào).

Truy vấn Cypher thực tế:

# Truy vấn chiều xuôi (outgoing)
MATCH (n:Norm) WHERE n.norm_number = $code
MATCH (n)-[r:CITES|IMPLEMENTS|SUPPLEMENTS|AMENDS|PARTIALLY_TERMINATES|TERMINATES|REFERS_TO|RELATED_TO]->(target:Norm)
RETURN type(r) AS rel_type, target.norm_number AS code, target.norm_type AS type, 'outgoing' AS direction

# Truy vấn chiều ngược (incoming)
MATCH (n:Norm) WHERE n.norm_number = $code
MATCH (source:Norm)-[r:CITES|IMPLEMENTS|SUPPLEMENTS|AMENDS|PARTIALLY_TERMINATES|TERMINATES|REFERS_TO|RELATED_TO]->(n)
RETURN type(r) AS rel_type, source.norm_number AS code, source.norm_type AS type, 'incoming' AS direction

Giải thích chi tiết truy vấn:

  1. Chiều xuôi (outgoing): Tìm các văn bản mà văn bản hiện tại trích dẫn (CITES), tham chiếu (REFERS_TO), sửa đổi (AMENDS) hoặc hướng dẫn (IMPLEMENTS).
  2. Chiều ngược (incoming): Tìm các văn bản mới hơn đang tác động lên văn bản hiện tại (ví dụ: Nghị định mới bãi bỏ một phần Nghị định cũ).
  3. Hỗ trợ đủ 8 mối quan hệ: Đảm bảo không bỏ sót bất kỳ mối liên kết vật lý nào giữa các Norm trong DB thực tế.
  4. Bảo vệ lỗi: Bao bọc bằng try-except trả về danh sách rỗng [] nếu kết nối Neo4j bị ngắt quãng.

Hàm: get_component_actions(comp_id)

Chức năng: Tìm kiếm lịch sử sửa đổi của một điều luật (Component) qua chuỗi hành động sửa đổi bắc cầu (đệ quy) hoặc danh sách điều khoản dẫn chiếu chéo (flat fallback).

  • Tham số đầu vào: comp_id (str): ID duy nhất của điều khoản (ví dụ: "127429__c59").
  • Dữ liệu trả về: Danh sách lịch sử hành động sửa đổi/dẫn chiếu (chứa amending_doc, action_type, action_text, depth).

Truy vấn Cypher thực tế:

# Cách 1: Duyệt đệ quy chuỗi sửa đổi (Action Node)
MATCH path = (a:Action)-[:APPLY_TO|HAS_ACTION*1..6]->(c:Component {comp_id: $comp_id})
OPTIONAL MATCH (a)-[:HAS_TEXTUNIT]->(t:TextUnit)
RETURN a.amending_doc_number AS amending_doc, 
       a.relation_type AS action_type, 
       t.accumulated_text AS action_text,
       t.unit_id AS unit_id,
       length(path) AS depth
ORDER BY depth ASC

# Cách 2 (Fallback phẳng): Dẫn chiếu chéo REFERS_TO
MATCH (c:Component {comp_id: $comp_id})-[r:REFERS_TO]->(target:Component)
OPTIONAL MATCH (target)-[:HAS_TEXTUNIT]->(t:TextUnit)
RETURN target.comp_id AS target_comp_id, 
       target.citation AS citation, 
       t.accumulated_text AS text

Giải thích chi tiết truy vấn:

  1. Duyệt đệ quy chuỗi sửa đổi: [:APPLY_TO|HAS_ACTION*1..6] cho phép đi xuyên qua nhiều tầng sửa đổi trung gian để tìm ra sửa đổi mới nhất (độ sâu lớn nhất). Chiều đi đúng chuẩn trong DB là từ node `Action` hướng tới `Component` qua quan hệ APPLY_TO.
  2. Sắp xếp theo độ sâu: ORDER BY depth ASC sắp xếp kết quả sửa đổi từ trực tiếp đến bắc cầu để Agent dễ theo dõi diễn tiến pháp lý.
  3. Fallback dẫn chiếu chéo: Nếu điều khoản không bị sửa đổi bởi luật nào (danh sách actions trống), hệ thống thực hiện tìm kiếm các điều khoản mà điều này dẫn chiếu tới (quan hệ REFERS_TO) để lấy nội dung bổ trợ.

Hàm: get_component_context(comp_id)

Chức năng: Lấy thông tin tiêu đề điều luật, cấp bậc của nó, và tìm văn bản gốc chứa nó để làm ngữ cảnh nền tảng cho Agent lập luận.

  • Tham số đầu vào: comp_id (str): ID duy nhất của điều khoản.
  • Dữ liệu trả về: Dictionary chứa thông tin norm (văn bản gốc) và component (tiêu đề, trích dẫn, cấp bậc).

Truy vấn Cypher thực tế:

# 1. Lấy văn bản gốc chứa Component
MATCH (n:Norm)-[:CONTAINS]->(c:Component {comp_id: $comp_id})
RETURN n.norm_number AS code, n.norm_type AS type LIMIT 1

# 2a. Thử tìm qua Version (Nâng cao)
MATCH (c:Component {comp_id: $comp_id})-[:HAS_VERSION]->(v:Version)
RETURN c.title_text AS title, c.citation AS citation, c.level AS level, v.version_id AS version_id LIMIT 1

# 2b. Fallback tìm trực tiếp Component (Phẳng)
MATCH (c:Component {comp_id: $comp_id})
RETURN c.title_text AS title, c.citation AS citation, c.level AS level LIMIT 1

Giải thích chi tiết truy vấn:

  1. Xác định văn bản gốc: Quan hệ CONTAINS liên kết trực tiếp từ Norm mẹ tới Component con cấp 1. Lấy số hiệu văn bản làm căn cứ nguồn.
  2. Fallback tương thích: Ưu tiên lấy thông tin điều khoản đi kèm ID phiên bản Version. Nếu DB bị thiếu dữ liệu Version, hệ thống tự động fallback sang truy vấn Component phẳng để không làm gián đoạn việc hiển thị dữ liệu của Agent.
  3. Bảo vệ lỗi: Trả về cấu trúc mặc định an toàn {"norm": {}, "component": {}} khi xảy ra lỗi kết nối.

3. Tool: web_search_mcp.py

Nằm tại đường dẫn: tools/web_search_mcp.py. Tệp tin này cung cấp khả năng tìm kiếm internet fallback khi CSDL GraphRAG nội bộ không chứa đủ ngữ cảnh hoặc cần thông tin cập nhật mới nhất.

Hàm: web_search_mcp(query, max_results, limit_to_trusted, search_depth)

Chức năng: Truy vấn Tavily Search API (qua HTTP REST Client tích hợp sẵn) để tìm kiếm thông tin bổ trợ. Mặc định tự động giới hạn tìm kiếm trong tập hợp các tên miền tin cậy và tự động **loại bỏ kết quả trùng lặp URL** để tối ưu hóa lượng token gửi đến Agent.

  • Tham số đầu vào:
    • query (str): Từ khóa tìm kiếm hoặc câu hỏi thô.
    • max_results (int, mặc định = 5): Số lượng trang kết quả tối đa cần lấy.
    • limit_to_trusted (bool, mặc định = True): Quyết định có lọc kết quả theo danh sách website uy tín hay không.
    • search_depth (str, mặc định = "basic"): Độ sâu tìm kiếm của Tavily (chấp nhận "basic" hoặc "advanced" cho tra cứu chuyên sâu).
  • Danh sách tên miền tin cậy (Trusted Domains):
    • vbpl.vn (Cơ sở dữ liệu quốc gia về văn bản pháp luật)
    • thuvienphapluat.vn (Cơ sở dữ liệu tư nhân lớn về luật)
    • chinhphu.vn (Cổng thông tin điện tử Chính phủ Việt Nam)
    • mof.gov.vn (Cổng thông tin Bộ Tài chính)
    • gdt.gov.vn (Cổng thông tin Tổng cục Thuế)
  • Dữ liệu trả về (Output Schema): Danh sách các dictionary chứa:
    • title (str): Tiêu đề của trang kết quả tìm kiếm.
    • link (str): Đường dẫn liên kết nguồn (URL).
    • snippet (str): Nội dung tóm tắt ngắn từ trang web.
  • Cấu hình Proxy (Tùy chọn):
    • Hệ thống tự động đọc biến môi trường WEB_SEARCH_PROXY từ tệp .env (Ví dụ: http://user:pass@ip:port).
    • Nếu giá trị này trống hoặc không tồn tại, hệ thống tự động fallback sử dụng kết nối mạng trực tiếp, bảo đảm tính tiện dụng khi phát triển local.

Cơ chế Fallback & Tự phản tỉnh (Self-Reflection):

  1. Fallback 1 (Khi lỗi kết nối hoặc rate limit lọc): Nếu việc truy vấn kèm bộ lọc bị Tavily từ chối hoặc lỗi mạng, hàm tự động bắt exception và thử lại bằng cách gọi đệ quy với limit_to_trusted=False để lấy kết quả tự do không giới hạn tên miền.
  2. Fallback 2 (Khi lọc không có kết quả): Nếu tìm kiếm có lọc chạy thành công nhưng không trả về bất kỳ kết quả nào, hệ thống tự động kích hoạt tìm kiếm tự do không giới hạn tên miền trên Tavily để đảm bảo Agent luôn có dữ liệu thô.
  3. Bảo vệ an toàn vòng lặp: Nếu cả hai lần tìm kiếm đều thất bại, hàm trả về danh sách rỗng [] và in cảnh báo log để tránh gây crash cho toàn bộ AgentStateGraph.
