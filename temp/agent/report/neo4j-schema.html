Chi tiết kiến trúc đồ thị tri thức pháp lý 5 tầng Node và lịch sử các thay đổi truy vấn Cypher.

1. Kiến trúc Đồ thị 5 Tầng Node

Tầng 1 Norm — Văn bản pháp luật (5,941 nodes)

8 quan hệ pháp lý: CITES · IMPLEMENTS · SUPPLEMENTS · AMENDS · PARTIALLY_TERMINATES · TERMINATES · REFERS_TO · RELATED_TO

↓ CONTAINS (17,238)

Phân cấp: Chương → Mục → Điều → Khoản → Điểm (CONTAINS 16,430)

↓ HAS_VERSION ↓ HAS_ACTION (235)
Tầng 3 Version

Phiên bản hiệu lực (fallback nếu rỗng)

Tầng 4 Action (235 nodes)

APPLY_TO → Component đích

↓ HAS_TEXTUNIT (29,112 + 235)
Tầng 5 TextUnit — Nội dung + Vector Embedding (29,347 nodes)

4. Vai trò Thực tế của search_vector_db.py

Hàm search_vector_db.py đóng vai trò là Cửa ngõ (Entry Point) kết nối Câu hỏi tự nhiên của Người dùng với Đồ thị tri thức (GraphRAG). Quá trình xử lý thực tế diễn ra như sau:

Bước 1: Tìm kiếm tương đồng Ngữ nghĩa (Semantic Lookup)
Người dùng gửi câu hỏi ngôn ngữ tự nhiên. Hệ thống sinh vector embedding và truy vấn trên textunit_embedding_index của Neo4j để tìm ra các đoạn văn bản luật (TextUnit) có ý nghĩa sát nhất.
Bước 2: Xác định Điểm neo Đồ thị (Graph Anchoring)
Nhờ cơ chế CONTAINS*1..6, câu truy vấn tự động JOIN ngược lên để lấy comp_id (ID điều khoản) và norm_number (Số hiệu luật). Đây là các điểm neo (anchor) vật lý giúp hệ thống biết bắt đầu duyệt từ đâu trên đồ thị.
Bước 3: Tra cứu Đồ thị bổ trợ (Graph Traversal)
Hệ thống chuyển tiếp comp_id sang search_legal_graph.py để duyệt đệ quy tìm chuỗi sửa đổi (Action) hoặc chuyển tiếp norm_number sang get_norm_relations để tìm văn bản liên kết chéo.

Cypher Query thực tế

CALL db.index.vector.queryNodes('textunit_embedding_index', $limit, $query_vector)
YIELD node, score
OPTIONAL MATCH (c:Component)-[:HAS_TEXTUNIT]->(node)
OPTIONAL MATCH (n:Norm)-[:CONTAINS*1..6]->(c)
RETURN node.unit_id AS unit_id, 
       node.accumulated_text AS accumulated_text, 
       c.citation AS citation,
       c.comp_id AS comp_id,
       n.norm_number AS norm_number,
       n.norm_type AS norm_type,
       score

5. Cơ chế Duyệt Đệ quy Chuỗi Sửa đổi

Khi Điều khoản A bị sửa đổi bởi Điều khoản B, và Điều khoản B tiếp tục bị sửa đổi bởi Điều khoản C, đồ thị liên kết đệ quy qua chuỗi Action → APPLY_TO → Component → HAS_ACTION → Action:

Component C
Điều khoản Luật C
→ HAS_ACTION →
Action C
Sửa đổi B
↓ APPLY_TO (depth = 3)
Component B
Điều khoản Luật B
→ HAS_ACTION →
Action B
Sửa đổi A
↓ APPLY_TO (depth = 1)
Component A
Điều khoản gốc Luật A (Điểm tra cứu)

Cypher Query

MATCH path = (a:Action)-[:APPLY_TO|HAS_ACTION*1..6]->(c:Component {comp_id: $comp_id})
OPTIONAL MATCH (a)-[:HAS_TEXTUNIT]->(t:TextUnit)
RETURN a.amending_doc_number AS amending_doc,
       a.relation_type AS action_type,
       t.accumulated_text AS action_text,
       t.unit_id AS unit_id,
       length(path) AS depth
ORDER BY depth ASC

• depth = 1: Sửa đổi trực tiếp   • depth = 3: Sửa đổi bắc cầu   • ORDER BY depth ASC: Từ cũ → mới
